Schema for Building Pruning State-of-the-Art Tables

The state-of-the-art section should normalize context before ranking results. The four stages below define a reporting protocol that makes pruning tables defensible, especially when different papers mix sparsity, speedup, recovery, and hardware claims.

Stage 1

Normalize Model and Task Context

Establish whether two results are even comparable before examining the pruning algorithm itself.

Mandatory Context Fields
1
Model family and scale: BERT Base, LLaMA-7B, Mistral-7B, and similar identifiers must remain explicit.
2
Task regime: language modeling perplexity, downstream classification, extractive QA, or zero-shot evaluation.
3
Sequence length and benchmark set: these directly affect measured cost and quality.
4
Dense baseline: every pruned result must reference the exact dense model used as its starting point.
Stage 2

Normalize the Compression Formulation

The pruning result should be recorded together with the structural assumptions that generated it.

Method Definition
1
Pruning unit: weight, N:M group, head, FFN channel, block, layer, or token.
2
Topology: unstructured, semi-structured, structured, or heterogeneous structured search.
3
Optimization signal: magnitude, activation-aware, first-order, second-order, proxy-based, or evolutionary.
4
Target: sparsity percentage, model-size ratio, or explicit speedup request.
Stage 3

Record Calibration, Recovery, and Deployment

Report the cost required to obtain the final model, not only the quality of the final mask.

Overhead Fields
1
Calibration budget: number of samples, tokens, or sequences used for saliency estimation.
2
Recovery stage: none, LoRA recovery, distillation, or full fine-tuning.
3
Hardware context: GPU class, runtime, sparse kernel availability, and whether the speed claim is simulated or measured.
4
Search cost: for proxy or evolutionary methods, the search budget is part of the method overhead.
Stage 4

Report Outputs in Regime-Specific Tables

Final comparison should preserve the metric that corresponds to the method's true deployment claim.

Output Fields
1
Quality: perplexity, downstream accuracy, task average, or F1, with dense baseline retained in the same table.
2
Compression: retained parameters, removed structures, or effective speedup.
3
Runtime: measured latency and throughput whenever acceleration is part of the claim.
4
Table placement: insert the result into the correct regime table rather than a single mixed ranking.
Recommended use in the pruning chapter: place this reporting schema immediately before the state-of-the-art tables, then group the empirical section into unstructured and N:M methods, structured pruning with real architectural removal, and search-driven hardware-aware pruning. That ordering makes the tables read as comparable evidence rather than as a flat survey list.
